## 一、项目概述

本项目基于 **昇腾310B（OrangePi AIpro）** 开发板，利用 **MobileNet-SSD** 轻量级检测模型在 NPU 上进行实时目标检测，结合 **DeepSORT** 跟踪算法（卡尔曼滤波 + 匈牙利算法）实现多目标轨迹追踪。检测与跟踪结果通过串口发送至 **STM32F103C8T6**，驱动二轴舵机云台随目标转动，使目标始终保持在画面中心区域。

本项目为**二次开发**性质——开发环境（CANN 8.0）已由小组成员搭建完成，基础代码（检测推理与跟踪框架）已由老师提供并烧录至开发板。

开发策略采用**分阶段推进**：

> **第一阶段（优先级高）**：集中精力把基础的**检测+跟踪**跑通、调稳，确保跟踪 ID 稳定、轨迹连续，达到中等到良好的考核标准。
> **第二阶段（锦上添花）**：在基础功能稳固的前提下，再推进 STM32 舵机云台控制，冲击优秀。

### 5.3 二次开发工作内容（分阶段推进）

#### 第一阶段（优先级高）：检测+跟踪基础功能

> **目标**：跑通并调稳检测+跟踪链路，确保单帧检测准确、连续帧跟踪ID稳定。

**步骤1：三人共同理解代码（6/30）**

1. 吴铭帆主讲 `detection_app.py`：图像预处理（resize、normalize）→ 加载OM模型 → NPU推理 → 结果解码 → NMS
2. 梁乐天主（参）讲 `tracking_app.py`：卡尔曼滤波预测、IOU 矩阵计算、匈牙利算法匹配、轨迹创建/更新/删除
3. 张泽滨辅助理解串口通信部分：昇腾端串口发送代码、通信协议（帧格式、波特率等）
4. **三人各自跑一遍程序**，记录当前检测效果和跟踪表现

**步骤2：检测调优（吴铭帆，7/1）**

1. 调整 `--score-threshold`（默认0.5，尝试0.3~0.7区间）
2. 调整 `--nms-threshold`（默认0.5，尝试0.4~0.6）
3. 测试不同光照、不同距离下的检测效果
4. 选定最优参数组合

**步骤3：跟踪调优（梁乐天，7/1~7/2）**

1. 调整 `--track-max-age`（轨迹最大丢失帧数，默认30，尝试60/120）
2. 调整 `--track-min-hits`（轨迹最少命中次数，默认3，尝试1/5）
3. 调整 `--track-iou-threshold`（IOU关联阈值，默认0.3，尝试0.4/0.5）
4. 调整 `--track-center-distance-threshold`（中心距离补充匹配阈值）
5. 调整 `--track-size-smoothing` 和 `--track-score-smoothing`（框大小和分数平滑）
6. 测试 `--track-classes person` 只跟踪行人，降低干扰
7. **记录最优参数组合**，确保ID稳定后再进入第二阶段

**步骤4：基础验证（全体，7/2上午）**

1. 在摄像头实时场景下连续运行5分钟以上
2. 统计ID切换次数、跟踪丢失情况
3. 确保达到「可演示」状态

---

#### 第二阶段（锦上添花）：STM32云台控制 + 全链路联调

> **目标**：在基础跟踪稳定的前提下，增加云台随动功能，冲击优秀。

**步骤5：STM32云台控制开发（张泽滨，7/2~7/4）**

1. **硬件连接**
   - SG90 舵机信号线 → STM32 TIM PWM 输出引脚（两路：水平+俯仰）
   - ST-Link → STM32 SWD 调试接口
   - 昇腾310B UART TX → STM32 USART RX（通过 USB转TTL 模块）
2. **PWM驱动**：配置 TIM 产生 50Hz PWM，脉宽 0.5ms~2.5ms 对应 0°~180°
3. **串口接收**：USART 中断接收昇腾310B发送的目标坐标 `(x, y)`
   - 通信协议：帧头(0xAA) + 坐标x + 坐标y + 校验 + 帧尾(0x55)
   - 波特率：115200
4. **PID控制**：以画面中心 `(cx, cy)` 为期望位置，目标坐标偏差为误差输入，输出 PWM 控制量
5. **单独测试**：先用上位机模拟坐标发送，验证舵机响应正确

**步骤6：全链路联调（全体，7/4~7/6）**

1. **协议对齐**：确认昇腾端串口发送格式与 STM32 接收解析一致
2. **闭环测试**：摄像头 → 检测 → 跟踪 → 串口 → STM32 → 舵机云台，完整链路验证
3. **延迟优化**：调整推理帧率、串口波特率、PID 参数整定，减少跟踪滞后
4. **边界测试**：目标快速移动、目标短暂离开画面后重新出现、多目标交叉场景

## 六、风险管理

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 老师提供的代码存在 Bug 或不完整 | 低 | 高 | 小组先全面阅读理解代码，记录疑问向老师确认 |
| 跟踪ID频繁跳变 | 中 | **高** | **第一阶段重点解决**：调整卡尔曼参数、IOU匹配阈值、max_age，使用--track-classes缩小干扰 |
| 基础检测跟踪未调稳就急于做云台 | 中 | 高 | **策略风险**：严格按分阶段推进，基础功能未达标前不进入第二阶段 |
| 云台响应滞后（PID 参数不当） | 中 | 中 | 第二阶段再处理：从保守参数开始，逐步整定 |
| 舵机抖动或过冲 | 中 | 中 | 第二阶段再处理：加入输出限幅、死区控制、低通滤波 |
| 串口通信丢包或数据错位 | 低 | 中 | 第二阶段再处理：增加帧头帧尾校验，超时重传机制 |
| 团队成员时间冲突 | 中 | 中 | 预留缓冲时间，关键节点集体攻关 |
